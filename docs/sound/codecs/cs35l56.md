## SUMMARY

The CS35L56 is a smart amplifier rather than a plain DAC codec, and two concrete differences appear in its ASoC registration. The [`cs35l56_dai`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L668) array declares a capture direction alongside playback, and the capture path carries the current/voltage measurement (IV sense) tapped at the speaker terminals rather than a microphone. The component probe loads protection firmware onto the on-chip DSP and applies per-speaker calibration coefficients, so audio is gated until that firmware is present. Over SoundWire the part splits the two directions across separate DAIs, [`cs35l56-sdw1`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L691) for the [`SDW1 Playback`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L694) speaker stream and [`cs35l56-sdw1c`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L704) for the [`SDW1 Capture`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L707) feedback stream, both pointing at one shared [`cs35l56_sdw_dai_ops`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L660) function pointer struct. The third DAI, [`cs35l56-asp1`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L670), is the I2S/TDM alternative used when the part is wired to a serial port, and it is not part of the SoundWire path.

The codec binds to the SoundWire core as a peripheral through [`cs35l56_sdw_ops`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L407), a [`struct sdw_slave_ops`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L616) filling [`read_prop`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L616), [`interrupt_callback`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L616), and [`update_status`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L616), and reaches its register space through the [`cs35l56_regmap_sdw`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-shared.c#L1872) map. The DAI [`hw_params`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) op [`cs35l56_sdw_dai_hw_params()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L587) attaches the peripheral to a manager-allocated SoundWire stream with [`sdw_stream_add_slave()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/stream.c#L2117), selecting [`CS35L56_SDW1_PLAYBACK_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L311) (1) for the speaker stream and [`CS35L56_SDW1_CAPTURE_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L312) (3) for the IV-sense stream, and [`cs35l56_sdw_dai_hw_free()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L638) detaches it with [`sdw_stream_remove_slave()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/stream.c#L2228).

The DSP firmware is managed by the [`struct wm_adsp`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/wm_adsp.h#L27) helper, whose [`cs_dsp`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/firmware/cirrus/cs_dsp.h#L150) member is the first field of [`struct cs35l56_private`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.h#L32) so the codec object can be cast to it. ## SPECIFICATIONS

The CS35L56 register model and function enumeration follow the MIPI SoundWire Device Class for Audio (SDCA) layout for an amplifier with protection features, and the SoundWire bus itself follows the MIPI SoundWire (and DisCo for SoundWire) specification, which on x86-64 ACPI platforms drives peripheral enumeration from the ACPI namespace. The part is enumerated as a SoundWire peripheral whose manufacturer id is 0x01FA, matched by the [`cs35l56_sdw_id`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L578) table. Per-speaker calibration is read from a UEFI variable that a manufacturing process or a prior Windows boot wrote, and the protection firmware files are distributed through the linux-firmware repository. The register space is a flat 32-bit address space reaching into DSP1 program/data memory, described to regmap by [`cs35l56_regmap_sdw`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-shared.c#L1872) with [`max_register`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/regmap.h#L408) at [`CS35L56_DSP1_PMEM_5114`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L121). The MIPI specifications are membership-gated, so the layout here is described from the kernel source rather than from those documents.

- MIPI SoundWire Device Class for Audio (SDCA): Function Type Smart Amplifier (amplifier with protection features)
- MIPI SoundWire Specification: peripheral attach, data ports, and stream prepare
- UEFI Specification: GetVariable for the per-amplifier calibration variable store

## INTERFACES

The CS35L56 presents two interfaces. Toward ASoC it is a [`struct snd_soc_dai_ops`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) on each SoundWire DAI, narrowed to the ops a peripheral needs to join a manager-allocated stream, each reached through a [`snd_soc_dai_*()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/soc-dai.c) wrapper that treats an absent callback as success. ## DETAILS

### The playback DAI and the IV-sense capture DAI

A plain DAC codec declares only a playback direction, since it has nothing to send back. The CS35L56 declares both, and the capture direction is the current/voltage (IV) feedback the protection firmware consumes. Over SoundWire the part uses two separate DAIs, one playback-only and one capture-only, both pointing at the same ops:

```c
/* sound/soc/codecs/cs35l56.c:691 */
	{
		.name = "cs35l56-sdw1",
		.id = 1,
		.playback = {
			.stream_name = "SDW1 Playback",
			.channels_min = 1,
			.channels_max = 2,
			.rates = CS35L56_RATES,
			.formats = CS35L56_RX_FORMATS,
		},
		.symmetric_rate = 1,
		.ops = &cs35l56_sdw_dai_ops,
	},
	{
		.name = "cs35l56-sdw1c",
		.id = 2,
		.capture = {
			.stream_name = "SDW1 Capture",
			.channels_min = 1,
			.channels_max = 4,
			.rates = CS35L56_RATES,
			.formats = CS35L56_TX_FORMATS,
		},
		.symmetric_rate = 1,
		.ops = &cs35l56_sdw_dai_ops,
	},
```

The [`SDW1 Playback`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L694) stream is the audio the host sends to the speaker, and the [`SDW1 Capture`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L707) stream is the up-to-four-channel measurement the amplifier returns. The capture channel count exceeds the stereo playback because the amplifier can place several monitor signals (measured current, measured voltage, and DSP state) on the feedback port. The [`cs35l56-asp1`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L670) entry above these two is the I2S/TDM alternative used when the part is wired to a serial port rather than SoundWire, and it points at a different [`cs35l56_ops`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L686) struct.

```
    Two SoundWire DAIs: speaker playback and IV-sense capture
    ─────────────────────────────────────────────────────────

    DAI driver       dir       channels   stream name      formats
    ───────────────  ────────  ─────────  ───────────────  ─────────────
    cs35l56-sdw1     playback  1 .. 2     "SDW1 Playback"  CS35L56_RX_*
    (id 1)           ──▶ amp
    cs35l56-sdw1c    capture   1 .. 4     "SDW1 Capture"   CS35L56_TX_*
    (id 2)           ◀── amp

    both DAIs share .ops = cs35l56_sdw_dai_ops
    capture carries up to 4 monitor channels (I, V, DSP state);
    playback is stereo to the boosted speaker drive
```

### The component driver the amp registers under

The CS35L56 component driver carries the DAPM graph that routes the SoundWire playback port to the DSP and the DSP feedback to the SoundWire capture port, plus the [`set_bias_level`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-component.h#L67) hook that stalls audio until firmware patching finishes:

```c
/* sound/soc/codecs/cs35l56.c:1353 */
static const struct snd_soc_component_driver soc_component_dev_cs35l56 = {
	.probe = cs35l56_component_probe,
	.remove = cs35l56_component_remove,

	.dapm_widgets = cs35l56_dapm_widgets,
	.num_dapm_widgets = ARRAY_SIZE(cs35l56_dapm_widgets),
	.dapm_routes = cs35l56_audio_map,
	.num_dapm_routes = ARRAY_SIZE(cs35l56_audio_map),

	.set_bias_level = cs35l56_set_bias_level,

	.suspend_bias_off = 1, /* see cs35l56_system_resume() */
};
```

[`cs35l56_common_probe()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L1795) is the transport-independent probe shared by the SoundWire, I2C, and SPI front ends, and the SoundWire front end [`cs35l56_sdw_probe()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L508) is one of its three callers. It runs [`cs35l56_dsp_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L1534) to build the DSP wrapper and work queue, then registers the three DAIs under the component:

```c
/* sound/soc/codecs/cs35l56.c:1860 */
	ret = cs35l56_dsp_init(cs35l56);
	if (ret < 0) {
		dev_err_probe(cs35l56->base.dev, ret, "DSP init failed\n");
		goto err;
	}

	ret = devm_snd_soc_register_component(cs35l56->base.dev,
					      &soc_component_dev_cs35l56,
					      cs35l56_dai, ARRAY_SIZE(cs35l56_dai));
	if (ret < 0) {
		dev_err_probe(cs35l56->base.dev, ret, "Register codec failed\n");
		goto err;
	}
```

### The DAI ops attach the codec to a SoundWire stream

The CS35L56 implements the generic SoundWire DAI pattern. The [`set_stream`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) op stores the manager-allocated [`struct sdw_stream_runtime`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L930) handle as the DAI [`dma_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L425), and [`hw_params`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) reads it back and joins the peripheral to that stream. The ops set five callbacks:

```c
/* sound/soc/codecs/cs35l56.c:660 */
static const struct snd_soc_dai_ops cs35l56_sdw_dai_ops = {
	.set_tdm_slot = cs35l56_sdw_dai_set_tdm_slot,
	.shutdown = cs35l56_sdw_dai_shutdown,
	.hw_params = cs35l56_sdw_dai_hw_params,
	.hw_free = cs35l56_sdw_dai_hw_free,
	.set_stream = cs35l56_sdw_dai_set_stream,
};
```

[`cs35l56_sdw_dai_set_stream()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L652) stashes the handle the SoundWire manager passes down so the later [`hw_params`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) call can recover it through [`snd_soc_dai_get_dma_data()`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L497):

```c
/* sound/soc/codecs/cs35l56.c:652 */
static int cs35l56_sdw_dai_set_stream(struct snd_soc_dai *dai,
				      void *sdw_stream, int direction)
{
	snd_soc_dai_dma_data_set(dai, direction, sdw_stream);

	return 0;
}
```

[`cs35l56_sdw_dai_hw_params()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L587) refuses to run before the part is initialised, recovers the stream handle, builds the stream and port configuration, and selects the playback or capture data port by stream direction:

```c
/* sound/soc/codecs/cs35l56.c:598 */
	struct cs35l56_private *cs35l56 = snd_soc_component_get_drvdata(dai->component);
	struct sdw_stream_runtime *sdw_stream = snd_soc_dai_get_dma_data(dai, substream);
	struct sdw_stream_config sconfig;
	struct sdw_port_config pconfig;
	int ret;

	dev_dbg(cs35l56->base.dev, "%s: rate %d\n", __func__, params_rate(params));

	if (!cs35l56->base.init_done)
		return -ENODEV;

	if (!sdw_stream)
		return -EINVAL;

	memset(&sconfig, 0, sizeof(sconfig));
	memset(&pconfig, 0, sizeof(pconfig));

	sconfig.frame_rate = params_rate(params);
	sconfig.bps = snd_pcm_format_width(params_format(params));

	if (substream->stream == SNDRV_PCM_STREAM_PLAYBACK) {
		sconfig.direction = SDW_DATA_DIR_RX;
		pconfig.num = CS35L56_SDW1_PLAYBACK_PORT;
		pconfig.ch_mask = cs35l56->rx_mask;
	} else {
		sconfig.direction = SDW_DATA_DIR_TX;
		pconfig.num = CS35L56_SDW1_CAPTURE_PORT;
		pconfig.ch_mask = cs35l56->tx_mask;
	}
```

The playback stream uses [`SDW_DATA_DIR_RX`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L173) into data port [`CS35L56_SDW1_PLAYBACK_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L311) (1), and the IV-sense capture stream uses [`SDW_DATA_DIR_TX`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L174) out of [`CS35L56_SDW1_CAPTURE_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L312) (3). The direction names are from the peripheral's point of view, so RX is the speaker stream arriving at the amp and TX is the feedback leaving it. After computing the channel mask, the op joins the stream:

```c
/* sound/soc/codecs/cs35l56.c:628 */
	ret = sdw_stream_add_slave(cs35l56->sdw_peripheral, &sconfig, &pconfig,
				   1, sdw_stream);
	if (ret) {
		dev_err(dai->dev, "Failed to add sdw stream: %d\n", ret);
		return ret;
	}

	return 0;
}
```

The channel masks that fill [`ch_mask`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L893) come from [`cs35l56_sdw_dai_set_tdm_slot()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L575), which the machine driver calls to assign the SoundWire channel slots. According to its comment the CPU-side and codec-side senses are opposite, so it crosses tx into [`rx_mask`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.h#L32) and rx into [`tx_mask`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.h#L32):

```c
/* sound/soc/codecs/cs35l56.c:575 */
static int cs35l56_sdw_dai_set_tdm_slot(struct snd_soc_dai *dai, unsigned int tx_mask,
					unsigned int rx_mask, int slots, int slot_width)
{
	struct cs35l56_private *cs35l56 = snd_soc_component_get_drvdata(dai->component);

	/* rx/tx are from point of view of the CPU end so opposite to our rx/tx */
	cs35l56->rx_mask = tx_mask;
	cs35l56->tx_mask = rx_mask;

	return 0;
}
```

[`cs35l56_sdw_dai_hw_free()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L638) tears the stream membership down again with [`sdw_stream_remove_slave()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/stream.c#L2228):

```c
/* sound/soc/codecs/cs35l56.c:638 */

				   struct snd_soc_dai *dai)
{
	struct cs35l56_private *cs35l56 = snd_soc_component_get_drvdata(dai->component);
	struct sdw_stream_runtime *sdw_stream = snd_soc_dai_get_dma_data(dai, substream);

	if (!cs35l56->sdw_peripheral)
		return -EINVAL;

	sdw_stream_remove_slave(cs35l56->sdw_peripheral, sdw_stream);

	return 0;
}
```

The setup op turns the PCM stream direction into the SoundWire RX or TX setting, the playback or capture data port (1 or 3), and the channel mask that the slot setter crossed in beforehand:

```
    hw_params: PCM direction selects port, SDW dir, and channel mask
    ────────────────────────────────────────────────────────────────

    PCM stream            SDW direction      data port              ch_mask
    ────────────────────  ─────────────────  ─────────────────────  ───────
    SNDRV_..._PLAYBACK    SDW_DATA_DIR_RX    ..._SDW1_PLAYBACK_PORT  rx_mask
       (speaker in)                          (1)
    SNDRV_..._CAPTURE     SDW_DATA_DIR_TX    ..._SDW1_CAPTURE_PORT   tx_mask
       (IV sense out)                        (3)

    masks are set earlier by set_tdm_slot, which crosses CPU senses:
       cpu tx_mask ──▶ codec rx_mask        cpu rx_mask ──▶ codec tx_mask
    then sdw_stream_add_slave joins the peripheral to the stream
```

[`cs35l56_sdw_update_status()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L369) runs the one-time init the first time the manager reports the peripheral attached, which is how SoundWire enumeration drives codec setup rather than a synchronous probe. It only calls [`cs35l56_sdw_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L254) when the part is not yet initialised or is mid-soft-reset:

```c
/* sound/soc/codecs/cs35l56-sdw.c:369 */
static int cs35l56_sdw_update_status(struct sdw_slave *peripheral,
				     enum sdw_slave_status status)
{
	struct cs35l56_private *cs35l56 = dev_get_drvdata(&peripheral->dev);

	switch (status) {
	case SDW_SLAVE_ATTACHED:
		dev_dbg(cs35l56->base.dev, "%s: ATTACHED\n", __func__);
		if (cs35l56->sdw_attached)
			break;

		if (!cs35l56->base.init_done || cs35l56->soft_resetting)
			cs35l56_sdw_init(peripheral);

		cs35l56->sdw_attached = true;
		break;
	case SDW_SLAVE_UNATTACHED:
		dev_dbg(cs35l56->base.dev, "%s: UNATTACHED\n", __func__);
		cs35l56->sdw_attached = false;
		break;
	default:
		break;
	}

	return 0;
}
```

The protection interrupt is delivered through [`cs35l56_sdw_interrupt()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L289), which masks and clears the SoundWire general interrupt registers and queues [`cs35l56_sdw_irq_work()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L322) on the power-efficient queue. According to its comment the read of GEN_INT_STAT_1 is required by the SoundWire spec for the interrupt status bits to clear. The whole peripheral is registered through a [`struct sdw_driver`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L706) whose id table matches the Cirrus manufacturer id 0x01FA:

### The regmap reaches the amp over SoundWire

The CS35L56 does register access through a [`struct regmap`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/regmap.h#L35) rather than the bus directly. [`cs35l56_sdw_probe()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L508) allocates the private struct, picks this config by [`driver_data`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/mod_devicetable.h#L271), and binds it to the SoundWire peripheral through [`cs35l56_regmap_bus_sdw`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-sdw.c#L233), starting in cache-only mode until the device is enumerated:

```c
		return;

	pm_runtime_get_sync(cs35l56->base.dev);

	ret = cs35l56_read_prot_status(&cs35l56->base, &firmware_missing, &firmware_version);
	if (ret)
		goto err;
	...
	/*
	 * The firmware cannot be patched if it is already running from
	 * patch RAM. In this case the firmware files are versioned to
	 * match the running firmware version and will only contain
	 * tunings. We do not need to shutdown the firmware to apply
	 * tunings so can use the lower cost reinit sequence instead.
	 */
	if (!firmware_missing)
		cs35l56_reinit_patch(cs35l56);
	else
		cs35l56_patch(cs35l56, firmware_missing);

	cs35l56_log_tuning(&cs35l56->base, &cs35l56->dsp.cs_dsp);
err:
	pm_runtime_put_autosuspend(cs35l56->base.dev);
}
```

### Calibration comes from EFI keyed by silicon UID

Per-speaker calibration corrects the protection model for the individual voice coil, and the CS35L56 reads it from a UEFI variable written during manufacturing. [`cs35l56_get_calibration()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56-shared.c#L962) runs during [`cs35l56_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L1880) and looks the entry up by the part's silicon UID, skipping a secured part entirely and treating a missing entry as non-fatal:

```c
/* sound/soc/codecs/cs35l56-shared.c:962 */
int cs35l56_get_calibration(struct cs35l56_base *cs35l56_base)
{
	int ret;

	/* Driver can't apply calibration to a secured part, so skip */
	if (cs35l56_base->secured)
		return 0;

	ret = cs_amp_get_efi_calibration_data(cs35l56_base->dev,
					      cs35l56_base->silicon_uid,
					      cs35l56_base->cal_index,
					      &cs35l56_base->cal_data);

	/* Only return an error status if probe should be aborted */
	if ((ret == -ENOENT) || (ret == -EOVERFLOW))
		return 0;

	if (ret < 0)
		return ret;

	cs35l56_base->cal_data_valid = true;

	return 0;
}
```

[`cs_amp_get_efi_calibration_data()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c#L608) is the shared Cirrus amplifier-library entry, also used by the HDA-side [`cs35l41_get_calibration()`](https://elixir.bootlin.com/linux/v7.0/source/sound/hda/codecs/side-codecs/cs35l41_hda.c#L443). According to its kernel-doc it performs three kinds of lookup, a UID search, a UID search with array-index fallback, and an index-only fetch, so several identical amps each get their own entry:

```c
/* sound/soc/codecs/cs-amp-lib.c:608 */
int cs_amp_get_efi_calibration_data(struct device *dev, u64 target_uid, int amp_index,
				    struct cirrus_amp_cal_data *out_data)
{
	if (IS_ENABLED(CONFIG_EFI) || IS_ENABLED(CONFIG_SND_SOC_CS_AMP_LIB_TEST_HOOKS))
		return _cs_amp_get_efi_calibration_data(dev, target_uid, amp_index, out_data);
	else
		return -ENOENT;
}
```

The variable read happens in [`_cs_amp_get_efi_calibration_data()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c#L416), which calls [`cs_amp_get_cal_efi_buffer()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c#L332) to fetch the EFI variable named in the [`cs_amp_lib_cal_efivars`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c) table, validates its length, and then matches the requested UID against the non-zero entries in the array:

```c
/* sound/soc/codecs/cs-amp-lib.c:427 */
	if (target_uid) {
		for (i = 0; i < efi_data->count; ++i) {
			u64 cal_target = cs_amp_cal_target_u64(&efi_data->data[i]);

			/* Skip empty entries */
			if (!efi_data->data[i].calTime[0] && !efi_data->data[i].calTime[1])
				continue;

			/* Skip entries with unpopulated silicon ID */
			if (cal_target == 0)
				continue;

			if (cal_target == target_uid) {
				cal = &efi_data->data[i];
				break;
			}
		}
	}
```

The matched entry is copied into a [`struct cirrus_amp_cal_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs-amp-lib.h#L15), the same layout the manufacturing tool wrote into the UEFI variable:

```c
/* include/sound/cs-amp-lib.h:15 */
struct cirrus_amp_cal_data {
	u32 calTarget[2];
	u32 calTime[2];
	s8  calAmbient;
	u8  calStatus;
	u16 calR;
};
```

The lookup walks that one shared array entry by entry and copies out the first whose silicon UID matches, skipping the empty and unpopulated rows along the way:

```
    Matching one amp to its EFI calibration entry by silicon UID
    ────────────────────────────────────────────────────────────

    UEFI variable ─▶ efi_data->data[ ]   (one array, all amps)
    ┌─────────┬─────────┬─────────┬─────────┬─────────┐
    │ entry 0 │ entry 1 │ entry 2 │   ...   │ entry n │
    └─────────┴────┬────┴─────────┴─────────┴─────────┘
                   │ scan for target_uid = base.silicon_uid
                   ▼
    per entry, in order:
      calTime[0]==0 && calTime[1]==0  ─▶ empty,      skip
      cal_target == 0                 ─▶ unpopulated, skip
      cal_target == target_uid        ─▶ MATCH, copy out
                                          and stop

    fallback order (cs_amp_get_efi_calibration_data):
      UID search  ▶  UID search then array index  ▶  index-only fetch
    matched entry ─▶ struct cirrus_amp_cal_data base.cal_data
```

### Calibration is written into firmware controls

[`cs35l56_write_cal()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L718) runs from the patch and reinit paths once firmware is resident. It runs the DSP with [`wm_adsp_run()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/wm_adsp.c#L1089), writes the calibration fields to the firmware coefficient controls with [`cs_amp_write_cal_coeffs()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c#L210), then stops the DSP again with [`wm_adsp_stop()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/wm_adsp.c#L1097):

```c
/* sound/soc/codecs/cs35l56.c:718 */
static int cs35l56_write_cal(struct cs35l56_private *cs35l56)
{
	int ret;

	if (cs35l56->base.secured || !cs35l56->base.cal_data_valid)
		return -ENODATA;

	ret = wm_adsp_run(&cs35l56->dsp);
	if (ret)
		return ret;

	ret = cs_amp_write_cal_coeffs(&cs35l56->dsp.cs_dsp,
				      cs35l56->base.calibration_controls,
				      &cs35l56->base.cal_data);

	wm_adsp_stop(&cs35l56->dsp);

	if (ret == 0)
		dev_info(cs35l56->base.dev, "Calibration applied\n");

	return ret;
}
```

[`cs_amp_write_cal_coeffs()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs-amp-lib.c#L210) takes the [`struct cs_dsp`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/firmware/cirrus/cs_dsp.h#L150) directly, plus a [`struct cirrus_amp_cal_controls`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs-amp-lib.h#L38) naming the firmware controls each field is written to, so the same writer serves the SoundWire driver here and the HDA-side [`cs35l56_hda_apply_calibration()`](https://elixir.bootlin.com/linux/v7.0/source/sound/hda/codecs/side-codecs/cs35l56_hda.c#L547):

```c
/* sound/soc/codecs/cs-amp-lib.c:210 */

			    const struct cirrus_amp_cal_controls *controls,
```

### Bias-level transitions wait for the queued firmware

Because the firmware download runs asynchronously, the first audio start must not race ahead of it. [`cs35l56_set_bias_level()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L1330) is the [`set_bias_level`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-component.h#L67) hook, and on the BIAS_OFF to BIAS_STANDBY transition it blocks in [`cs35l56_wait_dsp_ready()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c) until the queued [`cs35l56_dsp_work()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/cs35l56.c#L858) has completed:

```c
/* sound/soc/codecs/cs35l56.c:1330 */
static int cs35l56_set_bias_level(struct snd_soc_component *component,
				  enum snd_soc_bias_level level)
{
	struct cs35l56_private *cs35l56 = snd_soc_component_get_drvdata(component);
	struct snd_soc_dapm_context *dapm = snd_soc_component_to_dapm(component);

	switch (level) {
	case SND_SOC_BIAS_STANDBY:
		/*
		 * Wait for patching to complete when transitioning from
		 * BIAS_OFF to BIAS_STANDBY
		 */
		if (snd_soc_dapm_get_bias_level(dapm) == SND_SOC_BIAS_OFF)
			cs35l56_wait_dsp_ready(cs35l56);

		break;
	default:
		break;
	}

	return 0;
}
```

This is the last gate before the stream the DAI ops joined actually drives the speaker. The DAI [`hw_params`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-dai.h#L269) op refuses to run before [`init_done`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L347) is set, the bias-level hook waits for the firmware download to finish, and only then does the boosted, calibrated, protection-firmware-gated audio reach the voice coil while the IV-sense capture stream returns the measured current and voltage that bound excursion and temperature.

```
    Bias ladder with the OFF ▶ STANDBY firmware gate
    ────────────────────────────────────────────────

    ┌──────────┐                      ┌──────────────┐
    │ BIAS_OFF │ ─ up ─▶              │ BIAS_STANDBY │
    └──────────┘                      └───────┬──────┘
         ▲     cs35l56_wait_dsp_ready         │
         │     blocks until the               ▼
         │     queued dsp_work        ┌──────────────┐
         │     has finished           │ BIAS_PREPARE │
         │                            └───────┬──────┘
         │                                    ▼
    ┌──────────┐                      ┌──────────────┐
    │ BIAS_OFF │ ◀ down               │   BIAS_ON    │
    └──────────┘                      └──────────────┘

    only BIAS_OFF ▶ BIAS_STANDBY gates on firmware;
    every other transition returns at once
```

### SoundWire smart-amplifier codec structure

A SoundWire smart speaker amplifier presents as one [`struct snd_soc_component`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-component.h#L207) whose DP1 sink playback DAI on [`CS35L56_SDW1_PLAYBACK_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L311) feeds the [`struct wm_adsp`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/wm_adsp.h#L27) boost and protection firmware driving the speaker, while a DP3 source capture DAI on [`CS35L56_SDW1_CAPTURE_PORT`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/cs35l56.h#L312) carries the IV-sense feedback back.

```
    SoundWire smart speaker amplifier codec
    ───────────────────────────────────────

       SoundWire bus (x86-64 ACPI, MIPI DisCo enumeration)
       ┌──────────────────────────────────────────────────────────┐
       │  manager: SOF/Intel SoundWire controller                 │
       └───────────────────────────────┬──────────────────────────┘
                                       │ peripheral DP ports
       ┌───────────────────────────────▼──────────────────────────┐
       │  smart amplifier codec  (struct snd_soc_component)       │
       │                                                          │
       │    DP1 sink ──▶ playback DAI ──▶ DSP boost + protection  │
       │       (speaker stream)               firmware            │
       │                                         │                │
       │                                         ▼                │
       │                                  boosted drive ──▶ SPK   │
       │                                         │                │
       │    DP3 src ◀── capture DAI ◀── IV sense (I and V tap)    │
       │       (feedback stream)            at the speaker output │
       └──────────────────────────────────────────────────────────┘

       speaker output drives the voice coil; the IV-sense capture
       returns measured current and voltage so the protection model
       bounds excursion and temperature
```
