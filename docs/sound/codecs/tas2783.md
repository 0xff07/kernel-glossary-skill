## SUMMARY

The TAS2783 driver is one C file. It builds on the generic SDCA layer ([`struct sdca_function_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1419), parsed by [`sdca_parse_function()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdca/sdca_functions.c#L2163)) for its register layout and power model, and on its own RCA firmware loader plus a UEFI calibration store for the speaker-protection coefficients. The ASoC face of the codec is [`soc_codec_driver_tasdevice`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1027), a [`struct snd_soc_component_driver`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc-component.h#L67) that names a control table [`tas2783_snd_controls`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L554), a widget array [`tas_dapm_widgets`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L856), and a route array [`tas_audio_map`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L868), registered with [`devm_snd_soc_register_component()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/soc-devres.c#L29) from [`tas_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1040). A single [`tas_dai_driver`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L985) entry named `tas2783-codec` carries both a [`playback`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L989) capability (the speaker stream) and a [`capture`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L996) capability (the current/voltage IV-sense feedback), both pointing at [`tas_dai_ops`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L978).

The codec reaches its registers through a SoundWire MBQ (Multi-Byte Quantity) regmap because its SDCA Controls span one, two, or four bytes; [`tas_sdw_probe()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1271) builds it with [`devm_regmap_init_sdw_mbq_cfg()`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/regmap.h#L1226) from [`tas_regmap`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L513) and [`tas2783_mbq_cfg`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L526), whose per-register byte width comes from [`tas2783_sdca_mbq_size()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L338). ## SPECIFICATIONS

The TAS2783 register model and Function enumeration follow the MIPI SoundWire Device Class for Audio (SDCA) and the MIPI SoundWire bus specification, parsed from the ACPI DisCo `_DSD` properties. The [`enum sdca_function_type`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L72) value [`SDCA_FUNCTION_TYPE_SMART_AMP`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L73) (0x01) is documented in the [`include/sound/sdca_function.h`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h) kerneldoc as "Amplifier with protection features", citing SDCA specification v1.0a Section 5.1.2, and the comment notes that the SIMPLE_AMP type next to it is "Subset of SmartAmp". The on-wire SDCA Control address is packed by [`SDW_SDCA_CTL`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw_registers.h#L335), whose bit table the comment above it labels "v1.2 device - SDCA address mapping". The TAS2783's own SDCA Entity and Control numbers ([`TAS2783_SDCA_ENT_PDE23`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L63), [`TAS2783_SDCA_CTL_REQ_POW_STATE`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L85)) are defined in [`sound/soc/codecs/tas2783.h`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h) from the part's SDCA description. Calibration data is read from a UEFI variable that a manufacturing process or a prior Windows boot wrote, keyed by a vendor GUID.

- MIPI SoundWire Device Class for Audio (SDCA), section 5.1.2: Function Types (Smart Amplifier, 0x01)
- MIPI SoundWire Specification: peripheral attach, data ports, stream prepare, and the Multi-Byte Quantity control transport
- UEFI Specification: GetVariable for the `SmartAmpCalibrationData` variable store

## DETAILS

### The generic SDCA Function the TAS2783 parses

An SDCA peripheral describes its internal audio topology as a set of DisCo properties under a per-Function ACPI child node, and the kernel reads that description into a [`struct sdca_function_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1419) so that a driver consumes the parsed data rather than the raw firmware. The TAS2783 reuses this generic layer. The parsed Function holds a back pointer to the short descriptor, an init-write table replayed at power-up, the Entity array, and the Function-level busy and reset timeouts:

```c
/* include/sound/sdca_function.h:1419 */
struct sdca_function_data {
	struct sdca_function_desc *desc;

	struct sdca_init_write *init_table;
	struct sdca_entity *entities;
	struct sdca_cluster *clusters;
	int num_init_table;
	int num_entities;
	int num_clusters;

	unsigned int busy_max_delay;
	unsigned int reset_max_delay;

	struct sdca_fdl_data fdl_data;
};
```

[`sdca_parse_function()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdca/sdca_functions.c#L2163) populates it. It records the [`desc`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1419) back pointer, reads the two Function-level delays, then calls the per-section finders in sequence:

```c
/* sound/soc/sdca/sdca_functions.c:2163 */
int sdca_parse_function(struct device *dev, struct sdw_slave *sdw,
			struct sdca_function_desc *function_desc,
			struct sdca_function_data *function)
{
	u32 tmp;
	int ret;

	function->desc = function_desc;

	ret = fwnode_property_read_u32(function_desc->node,
				       "mipi-sdca-function-busy-max-delay", &tmp);
	if (!ret)
		function->busy_max_delay = tmp;
	...
	ret = find_sdca_init_table(dev, function_desc->node, function);
	if (ret)
		return ret;

	ret = find_sdca_entities(dev, sdw, function_desc->node, function);
	if (ret)
		return ret;

	ret = find_sdca_connections(dev, function_desc->node, function);
	if (ret)
		return ret;

	ret = find_sdca_clusters(dev, function_desc->node, function);
	if (ret < 0)
		return ret;

	ret = find_sdca_filesets(dev, sdw, function_desc->node, function);
	if (ret)
		return ret;

	return 0;
}
```

Among the Entities a Smart Amp Function carries is a Power-Domain Entity, the SDCA block that gates power to the amplifier. Its parsed form is [`struct sdca_entity_pde`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1020), whose [`managed`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1020) array points at the Entities it powers and whose [`max_delay`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1020) array records how long each power-state transition takes:

```c
/* include/sound/sdca_function.h:1020 */
struct sdca_entity_pde {
	struct sdca_entity **managed;
	struct sdca_pde_delay *max_delay;
	int num_managed;
	int num_max_delay;
};
```

[`find_sdca_entity_pde()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdca/sdca_functions.c#L1213) fills [`max_delay`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1020) by reading the `mipi-sdca-powerdomain-transition-max-delay` list as (from-state, to-state, microseconds) triples, so a driver that requests a power state can wait the correct time before using the entity. The TAS2783's amplifier power-domain entity is [`TAS2783_SDCA_ENT_PDE23`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L63).

```
    Parsed SDCA Function the TAS2783 keeps in sa_func_data
    ──────────────────────────────────────────────────────

    ┌─────────────────────────────────────────────────────────┐
    │ struct sdca_function_data   (filled by sdca_parse_func) │
    │   desc ─────────▶ struct sdca_function_desc (back ptr)  │
    │   init_table[]    address/value writes replayed at boot │
    │   entities[] ───┐ num_entities                          │
    │   clusters[]    │ busy_max_delay / reset_max_delay      │
    └─────────────────┼───────────────────────────────────────┘
                      ▼  one entity is the Power-Domain Entity
    ┌─────────────────────────────────────────────────────────┐
    │ struct sdca_entity_pde   (TAS2783_SDCA_ENT_PDE23)       │
    │   managed[] ─────▶ entities this PDE powers             │
    │   max_delay[] ───▶ (from_state, to_state, microseconds) │
    └─────────────────────────────────────────────────────────┘

    sdca_data.function[] ─▶ sdca_parse_function ─▶ sa_func_data;
    tas_io_init replays init_table at attach
```

### The SMART_AMP type and the descriptor inventory

Every SDCA Function advertises a topology type, and the enumeration pass records it in the short [`struct sdca_function_desc`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca.h#L30) the peripheral keeps. The TAS2783 reads the [`type`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca.h#L30) field of each descriptor to find its amplifier Function:

```c
/* include/sound/sdca.h:30 */
struct sdca_function_desc {
	struct fwnode_handle *node;
	struct sdca_dev *func_dev;
	const char *name;
	u32 type;
	u8 adr;
};
```

The type codes are the [`enum sdca_function_type`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L72) values, whose kerneldoc names each audio role. The Smart Amp the TAS2783 implements is the first entry, distinct from the Simple Amp that has no protection firmware:

```c
/* include/sound/sdca_function.h:72 */
enum sdca_function_type {
	SDCA_FUNCTION_TYPE_SMART_AMP			= 0x01,
	SDCA_FUNCTION_TYPE_SIMPLE_AMP			= 0x02,
	SDCA_FUNCTION_TYPE_SMART_MIC			= 0x03,
	SDCA_FUNCTION_TYPE_SIMPLE_MIC			= 0x04,
	SDCA_FUNCTION_TYPE_SPEAKER_MIC			= 0x05,
	SDCA_FUNCTION_TYPE_UAJ				= 0x06,
	SDCA_FUNCTION_TYPE_RJ				= 0x07,
	SDCA_FUNCTION_TYPE_SIMPLE_JACK			= 0x08,
	SDCA_FUNCTION_TYPE_HID				= 0x0A,
	SDCA_FUNCTION_TYPE_COMPANION_AMP		= 0x0B,
	SDCA_FUNCTION_TYPE_IMP_DEF			= 0x1F,
};
```

According to the kerneldoc above the enum, these are "SDCA Function Types from SDCA specification v1.0a Section 5.1.2", and the [`SDCA_FUNCTION_TYPE_SMART_AMP`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L73) line is documented as "Amplifier with protection features". The descriptors are held in the per-peripheral [`struct sdca_device_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca.h#L47) cache embedded as [`slave->sdca_data`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L688), which SoundWire enumeration fills before any codec driver probes.

### The probe finds the Smart Amp Function and builds the MBQ regmap

[`tas_sdw_probe()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1271) runs when the SoundWire core binds the driver to a peripheral whose MIPI id matches [`tas_sdw_id`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1356) (manufacturer 0x0102, part 0x0000). It first reads the peripheral's properties, allocates the private state, then scans the cached descriptor inventory for [`SDCA_FUNCTION_TYPE_SMART_AMP`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L73):

```c
/* sound/soc/codecs/tas2783-sdw.c:1290 */
	i = -1;
	/* check if we have any SDCA function data available */
	if (peripheral->sdca_data.num_functions > 0) {
		dev_dbg(dev, "SDCA functions found: %d", peripheral->sdca_data.num_functions);

		/* Look for Smart Amp function type */
		for (i = 0; i < peripheral->sdca_data.num_functions; i++) {
			if (peripheral->sdca_data.function[i].type ==
			    SDCA_FUNCTION_TYPE_SMART_AMP) {
				dev_info(dev, "Found Smart Amp function at index %d", i);
				break;
			}
		}
	}
```

When the Function is found, the probe allocates a [`struct sdca_function_data`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1419), parses the matching descriptor into it, and stores the result so [`tas_io_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1136) can use the parsed init table later. A parse failure is non-fatal, the driver falls back to a hardcoded init sequence:

```c
/* sound/soc/codecs/tas2783-sdw.c:1313 */
		/* Parse the function */
		ret = sdca_parse_function(dev, peripheral,
					  &peripheral->sdca_data.function[i],
					  function_data);
		if (!ret)
			tas_dev->sa_func_data = function_data;
		else
			dev_warn(dev, "smartamp function parse failed:err%d, using defaults", ret);
```

The probe then builds the regmap and hands off to [`tas_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1040). Because the SDCA Controls are multi-byte, the TAS2783 uses an MBQ regmap, and starts it in cache-only mode until the part is attached:

```c
/* sound/soc/codecs/tas2783-sdw.c:1331 */
	regmap = devm_regmap_init_sdw_mbq_cfg(&peripheral->dev,
					      peripheral,
					      &tas_regmap,
					      &tas2783_mbq_cfg);
	if (IS_ERR(regmap))
		return dev_err_probe(dev, PTR_ERR(regmap),
				     "Failed devm_regmap_init_sdw.");

	/* keep in cache until the device is fully initialized */
	regcache_cache_only(regmap, true);
	tas_dev->regmap = regmap;
	return tas_init(tas_dev);
```

The per-register byte width comes from [`tas2783_sdca_mbq_size()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L338), which switches on the SDCA Control address built by [`SDW_SDCA_CTL`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw_registers.h#L335) and returns 1, 2, or 4 bytes, with 0 marking an unreadable register. The MBQ config [`tas2783_mbq_cfg`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L526) names that callback, and the base [`tas_regmap`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L513) declares the 32-bit register, 8-bit value geometry:

```c
/* sound/soc/codecs/tas2783-sdw.c:513 */
static const struct regmap_config tas_regmap = {
	.reg_bits = 32,
	.val_bits = 8,
	.readable_reg = tas2783_readable_register,
	.volatile_reg = tas2783_volatile_register,
	.reg_defaults = tas2783_reg_default,
	.num_reg_defaults = ARRAY_SIZE(tas2783_reg_default),
	.max_register = 0x41008000 + TASDEV_REG_SDW(0xa1, 0x60, 0x7f),
	.cache_type = REGCACHE_MAPLE,
	.use_single_read = true,
	.use_single_write = true,
};
```

The component driver carries the codec's controls, widgets, and routes:

A plain DAC codec declares only a playback direction. A smart amplifier declares both, because the capture direction is the current/voltage (IV) feedback the protection model consumes. The TAS2783 folds both into one DAI named `tas2783-codec`:

```c
/* sound/soc/codecs/tas2783-sdw.c:985 */

			.formats	= TAS2783_DEVICE_FORMATS,
		},
		.capture = {
			.stream_name	= "Capture",
			.channels_min	= 1,
			.channels_max	= 4,

			.formats	= TAS2783_DEVICE_FORMATS,

The DAPM routing makes the two directions concrete. The `ASI` playback widget feeds the FU21 and FU23 Feature Units, which drive the `SPK` output; the `DMIC` input (the IV-sense tap at the speaker terminals) feeds the `ASI OUT` capture widget:

```c
/* sound/soc/codecs/tas2783-sdw.c:868 */
static const struct snd_soc_dapm_route tas_audio_map[] = {
	{"FU21", NULL, "ASI"},
	{"SPK", NULL, "FU21"},
	{"FU23", NULL, "ASI"},
	{"SPK", NULL, "FU23"},
	{"ASI OUT", NULL, "DMIC"},
};
```

The FU21 and FU23 widgets are event-driven DACs; their event handlers write the SDCA mute Control [`TAS2783_SDCA_CTL_FU_MUTE`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L86) on power-up and power-down through [`sdw_write_no_pm()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/bus.c#L480) to the address [`SDW_SDCA_CTL`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw_registers.h#L335) packs for the Feature Unit.

```

    playback "Playback"                 capture "Capture"
    ──────────────────                  ─────────────────

       ASI ───┬──▶ FU21 ──┬──▶ SPK
              └──▶ FU23 ──┘

       DMIC ──▶ ASI OUT     (IV-sense tap at the speaker)

    FU21 and FU23 are event-driven DACs; their widget events

    capture stream that carries it back
```

### The DAI ops join the SoundWire stream behind a firmware gate

[`tas_dai_ops`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L978) wires four callbacks, the same SoundWire DAI pattern other peripheral codecs use:

```c
/* sound/soc/codecs/tas2783-sdw.c:978 */
static const struct snd_soc_dai_ops tas_dai_ops = {
	.hw_params	= tas_sdw_hw_params,
	.hw_free	= tas_sdw_pcm_hw_free,
	.set_stream	= tas_set_sdw_stream,
	.shutdown	= tas_sdw_shutdown,
};
```

[`tas_set_sdw_stream()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L876) stores the manager-allocated stream handle on the DAI as its dma_data, and [`tas_sdw_hw_params()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L893) reads it back to join the stream. The op refuses to run before firmware has downloaded, then reads the stored handle:

```c
/* sound/soc/codecs/tas2783-sdw.c:906 */
	if (!tas_dev->fw_dl_success) {
		dev_err(tas_dev->dev, "error playback without fw download");
		return -EINVAL;
	}

	sdw_stream = snd_soc_dai_get_dma_data(dai, substream);
	if (!sdw_stream)
		return -EINVAL;
```

It then requests the on power state of the PDE23 entity through the SDCA control address, retrying because power-on can transiently fail, and unlocks before joining the stream:

```c
/* sound/soc/codecs/tas2783-sdw.c:920 */
	mutex_lock(&tas_dev->pde_lock);
	/*
	 * Sometimes, there is error returned during power on.
	 * So added retry logic to ensure power on so that
	 * port prepare succeeds
	 */
	do {
		ret = regmap_write(tas_dev->regmap,
				   SDW_SDCA_CTL(1, TAS2783_SDCA_ENT_PDE23,
						TAS2783_SDCA_CTL_REQ_POW_STATE, 0),
						TAS2783_SDCA_POW_STATE_ON);
		if (!ret)
			break;
		usleep_range(2000, 2200);
	} while (retry--);
	mutex_unlock(&tas_dev->pde_lock);
	if (ret)
		return ret;

	/* SoundWire specific configuration */
	snd_sdw_params_to_config(substream, params,
				 &stream_config, &port_config);
	/* port 1 for playback */
	if (substream->stream == SNDRV_PCM_STREAM_PLAYBACK)
		port_config.num = 1;
	else
		port_config.num = 2;

	ret = sdw_stream_add_slave(sdw_peripheral,
				   &stream_config, &port_config, 1, sdw_stream);
```

The control address is the value [`SDW_SDCA_CTL(1, TAS2783_SDCA_ENT_PDE23, TAS2783_SDCA_CTL_REQ_POW_STATE, 0)`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw_registers.h#L335) computes from Function Number 1, the PDE23 Entity (0x0C), the requested-power-state Control selector (0x01), and channel 0. [`TAS2783_SDCA_POW_STATE_ON`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L92) is the active state (0). The playback stream uses port 1 and the IV-sense capture uses port 2. The matching [`tas_sdw_pcm_hw_free()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L956) removes the slave with [`sdw_stream_remove_slave()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/stream.c#L2228) and writes [`TAS2783_SDCA_POW_STATE_OFF`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783.h#L93) (3) back to the same PDE23 control.

```
    SDW_SDCA_CTL packs four selectors into one 32-bit address
    ─────────────────────────────────────────────────────────

    SDW_SDCA_CTL(func, entity, control-selector, control-number)
                       func  entity      ctl-sel      ctl-num
                         │      │           │            │
                         ▼      ▼           ▼            ▼
                      ┌─────┬───────┬───────────────┬─────────┐
                      │  1  │ PDE23 │ REQ_POW_STATE │    0    │
                      │     │(0x0C) │    (0x01)     │(channel)│
                      └─────┴───────┴───────────────┴─────────┘
                        one SDCA Control address (32-bit)

    value at that address: ON = 0 (PS0), OFF = 3 (PS3)
    hw_params requests ON before joining the stream;
    hw_free writes OFF.  port: playback ▶ 1, capture ▶ 2
    both refuse to run until fw_dl_success is set
```

[`tas_update_status()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1191) records the new status, clears [`hw_init`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L98) on detach, and on the first [`SDW_SLAVE_ATTACHED`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/soundwire/sdw.h#L96) brings the regmap out of cache-only mode and runs [`tas_io_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1136):

```c

		return 0;

Without this, the port fails to enter the prepared state resulting in no audio output." ### tas_io_init resets the part and requests RCA firmware

[`tas_io_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1136) runs once per attach. It software-resets the part, builds the RCA file name from the platform identity, and requests the firmware asynchronously, then blocks on a wait queue until the completion runs:

```c
/* sound/soc/codecs/tas2783-sdw.c:1148 */
	ret = regmap_write(tas_dev->regmap, TAS2783_SW_RESET, 0x1);
	if (ret) {
		dev_err(dev, "sw reset failed, err=%d", ret);
		return ret;
	}
	usleep_range(2000, 2200);

	tas_generate_fw_name(slave, tas_dev->rca_binaryname,
			     sizeof(tas_dev->rca_binaryname));

	ret = request_firmware_nowait(THIS_MODULE, FW_ACTION_UEVENT,
				      tas_dev->rca_binaryname, tas_dev->dev,
				      GFP_KERNEL, tas_dev, tas2783_fw_ready);
	...
	ret = wait_event_timeout(tas_dev->fw_wait, tas_dev->fw_dl_task_done,
				 msecs_to_jiffies(TIMEOUT_FW_DL_MS));
```

[`tas_generate_fw_name()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1111) walks the device's parents to find a PCI device, and when found names the file `<subsystem_device>-<link_id>-<unique_id>.bin`, falling back to `tas2783-<link_id>-<unique_id>.bin`. After the download completes, [`tas_io_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L1136) applies the init writes, preferring the parsed SDCA init table when a Function was parsed and the hardcoded sequence otherwise:

```c
/* sound/soc/codecs/tas2783-sdw.c:1174 */

			ret = sdca_regmap_write_init(dev, tas_dev->regmap,
						     tas_dev->sa_func_data);
		else
			ret = regmap_multi_reg_write(tas_dev->regmap, tas2783_init_seq,
						     ARRAY_SIZE(tas2783_init_seq));
```

[`sdca_regmap_write_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdca/sdca_regmap.c#L367) replays the [`init_table`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/sdca_function.h#L1419) that [`sdca_parse_function()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdca/sdca_functions.c#L2163) read from firmware, one [`regmap_write()`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/regmap.h) per address/value pair.

### tas2783_fw_ready downloads the firmware files

The RCA image is a header followed by a series of files, each with a destination address. [`tas2783_fw_ready()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L728) reads the header with [`tas_fw_read_hdr()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L704), checks that the stored size matches the firmware size, then walks the files and writes each one to its destination address over SoundWire:

```c
/* sound/soc/codecs/tas2783-sdw.c:767 */
	mutex_lock(&tas_dev->pde_lock);
	while (offset < (img_sz - FW_FL_HDR)) {
		offset += tas_fw_get_next_file(&buf[offset], file);
		dev_dbg(tas_dev->dev,
			"v=%d, fid=%d, ver=%d, len=%d, daddr=0x%x, fw=%p",
			file->vendor_id, file->file_id,
			file->version, file->length,
			file->dest_addr, file->fw_data);

		ret = sdw_nwrite_no_pm(tas_dev->sdw_peripheral,
				       file->dest_addr,
				       file->length,
				       file->fw_data);
		if (ret < 0) {
			dev_err(tas_dev->dev,
				"FW download failed: %d", ret);
			break;
		}
		cur_file++;
	}
	mutex_unlock(&tas_dev->pde_lock);
```

[`sdw_nwrite_no_pm()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/soundwire/bus.c#L468) writes a block of bytes to a SoundWire register address without taking a runtime-PM reference, which suits a bulk download where the caller already holds the part awake. After the last file, the callback applies calibration and signals the waiter:

```c
/* sound/soc/codecs/tas2783-sdw.c:789 */
	if (cur_file == 0) {
		dev_err(tas_dev->dev, "fw with no files");
		ret = -EINVAL;
	} else {
		tas2783_update_calibdata(tas_dev);
	}

out:
	if (!ret)
		tas_dev->fw_dl_success = true;
	tas_dev->fw_dl_task_done = true;
	wake_up(&tas_dev->fw_wait);
	if (fmw)
		release_firmware(fmw);
```

Setting [`fw_dl_success`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L102) is what later lets [`tas_sdw_hw_params()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L893) start a stream; until the download succeeds, every attempt to open the PCM returns `-EINVAL`.

```
    RCA firmware image walked by tas2783_fw_ready
    ─────────────────────────────────────────────

    ┌──────────────────────────────────────────┐
    │ RCA header  (FW_FL_HDR bytes)            │
    │   total image size, file count           │
    └──────────────────────────────────────────┘
                        │  then a sequence of files, repeated:
                        ▼
    ┌──────────────────────────────────────────┐
    │ file record  (tas_fw_get_next_file)      │
    │   vendor_id  file_id  version  length    │
    │   dest_addr  ─▶ destination in DSP space │
    │   fw_data    ─▶ block written there      │
    └──────────────────────────────────────────┘

    each fw_data block goes to its dest_addr over SoundWire
    via sdw_nwrite_no_pm(); loop until offset reaches
    img_sz - FW_FL_HDR.  after the last file:
    tas2783_update_calibdata, then fw_dl_success = true
```

### Calibration comes from a UEFI variable, matched per amp

Per-speaker calibration corrects the protection model for the individual voice coil, and the TAS2783 reads it from a UEFI variable that a manufacturing process or a prior Windows boot wrote. [`tas2783_update_calibdata()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L650) tries two variable names under a vendor GUID, querying the size first and then reading the data:

```c
/* sound/soc/codecs/tas2783-sdw.c:650 */
static s32 tas2783_update_calibdata(struct tas2783_prv *tas_dev)
{
	efi_guid_t efi_guid = TAS2783_CALI_GUID;
	u32 attr, i, *tmp_val;
	unsigned long size;
	s32 ret;
	efi_status_t status;
	static efi_char16_t efi_names[][32] = {
		L"SmartAmpCalibrationData", L"CALI_DATA"};
	...
	for (i = 0; i < ARRAY_SIZE(efi_names); i++) {
		size = 0;
		status = efi.get_variable(efi_names[i], &efi_guid, &attr,
					  &size, NULL);
		...
		if (status == EFI_BUFFER_TOO_SMALL) {
			status = efi.get_variable(efi_names[i], &efi_guid, &attr,
							&tas_dev->cali_data.read_sz,
							tas_dev->cali_data.data);
			...
		}
		if (status == EFI_SUCCESS)
			break;
	}
```

According to the comment, "In some cases, the calibration is performed in Windows, and data was saved in UEFI. Linux can access it." A missing variable is not an error; the function returns 0 and the part runs with firmware defaults. When the data is present, it is validated and then applied under [`calib_lock`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L88):

```c
/* sound/soc/codecs/tas2783-sdw.c:694 */
	mutex_lock(&tas_dev->calib_lock);
	ret = tas2783_validate_calibdata(tas_dev, tas_dev->cali_data.data,
					 tas_dev->cali_data.read_sz);
	if (!ret)
		tas2783_set_calib_params_to_device(tas_dev, tmp_val);
	mutex_unlock(&tas_dev->calib_lock);
```

[`tas2783_validate_calibdata()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L563) checks the magic number 2783, the speaker count, the computed size, and a CRC32 over the body. One UEFI blob holds calibration for every amp on the board, so [`tas2783_set_calib_params_to_device()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L615) walks the per-device records and applies only the one whose stored id matches this amp's SoundWire unique id:

```c
/* sound/soc/codecs/tas2783-sdw.c:624 */
	for (device_num = 0; device_num < dev_count; device_num++) {
		if (cali_data[offset] != tas_dev->sdw_peripheral->id.unique_id) {
			offset += TAS2783_CALIB_PARAMS;
			continue;
		}
		offset++;

		for (i = 0; i < ARRAY_SIZE(tas2783_cali_reg); i++) {
			reg_value = cali_data[offset + i];
			buf[0] = reg_value >> 24;
			buf[1] = reg_value >> 16;
			buf[2] = reg_value >> 8;
			buf[3] = reg_value & 0xff;
			regmap_bulk_write(tas_dev->regmap, tas2783_cali_reg[i],
					  buf, sizeof(u32));
		}
		break;
	}
```

The five target registers are [`tas2783_cali_reg`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/codecs/tas2783-sdw.c#L49), the calibrated R0, inverse R0, low-R0, power, and temperature-limit coefficients, each written as a four-byte big-endian value into the DSP register space.

```
    One UEFI blob, one record per amp, matched by unique_id
    ───────────────────────────────────────────────────────

    SmartAmpCalibrationData (or CALI_DATA), vendor GUID
    ┌──────────────┬──────────────┬──────────────┬─────────┐
    │ record id=A  │ record id=B  │ record id=C  │   ...   │
    │ + params     │ + params     │ + params     │         │
    └──────┬───────┴──────────────┴──────────────┴─────────┘
           │  for each record: cali_data[offset] == this
           │  amp's slave->id.unique_id ?
           ▼
        match ─▶ write tas2783_cali_reg[0..4], each a
                 4-byte big-endian value into DSP space:
                   R0,  inverse R0,  low-R0,  power,  temp-limit
        no match ─▶ advance offset += TAS2783_CALIB_PARAMS

    blob is validated first (magic 2783, speaker count, CRC32);
    a missing variable keeps the existing defaults
```

### One TAS2783 per speaker

When a laptop carries two TAS2783 amps, each enumerates as its own SoundWire peripheral with a distinct unique id, and the machine driver instantiates each as a separate codec on the card. The Intel SoundWire generic machine identifies the part through its [`codec_info_list`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdw_utils/soc_sdw_utils.c#L74) entry, which keyed on part_id 0x0000 sets the [`name_prefix`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdw_utils/soc_sdw_utils.c#L77) `tas2783`, names the `tas2783-codec` DAI, and points at the TI amp init [`asoc_sdw_ti_amp_init()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/sdw_utils/soc_sdw_ti_amp.c#L85):

```c
/* sound/soc/sdw_utils/soc_sdw_utils.c:74 */
struct asoc_sdw_codec_info codec_info_list[] = {
	{
		.part_id = 0x0000, /* TAS2783A */
		.name_prefix = "tas2783",
		.dais = {
			{
				.direction = {true, true},
				.dai_name = "tas2783-codec",
				.dai_type = SOC_SDW_DAI_TYPE_AMP,
				.dailink = {SOC_SDW_AMP_OUT_DAI_ID, SOC_SDW_AMP_IN_DAI_ID},
				.init = asoc_sdw_ti_amp_init,
				.rtd_init = asoc_sdw_ti_spk_rtd_init,
				...
```

[`create_sdw_dailink()`](https://elixir.bootlin.com/linux/v7.0/source/sound/soc/intel/boards/sof_sdw.c#L876) copies each endpoint's prefix into a [`struct snd_soc_codec_conf`](https://elixir.bootlin.com/linux/v7.0/source/include/sound/soc.h#L946), and ASoC prepends that prefix to every control name, so two amps that both expose `Amp Volume` and `Speaker Volume` become `tas2783 Amp Volume` and so on under distinct prefixes, keeping the per-speaker controls separate in userspace.
